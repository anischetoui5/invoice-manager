#####################################################################
# 1. UML STYLE — USE CASE DIAGRAMS
#####################################################################

## 1.1 Core notation

- System boundary: one rectangle per sprint, titled with the sprint
  scope. ALL use case ellipses live inside the rectangle. ALL actors
  live outside.
- Actor: standard UML stick figure + label. Human actors on the LEFT
  of the boundary. External systems on the RIGHT.
- Use case: ellipse with a short verb-noun label (max ~4 words).
  IDs (UC-X.Y) appear in this document for cross-reference only —
  NEVER inside the ellipse on the rendered diagram.
- Association (actor ↔ use case): plain solid line, no arrowhead.
- «include»: dashed arrow FROM the base UC TO the included UC, labeled
  «include». The included UC executes unconditionally every time the
  base UC executes.
- «extend»: dashed arrow FROM the extending UC TO the base UC, labeled
  «extend». Optionally annotate with the extension condition in square
  brackets on the arrow. The extending UC executes only under that
  specific condition.
- Use case generalization: solid line with hollow triangle, FROM child
  UC TO parent UC. Use only when one UC is truly a specialization of
  another — never as a substitute for «include»/«extend».

## 1.2 Actor naming — exact labels used in this document

Human actors (left side of boundary):
  User    — unauthenticated person (Sprint 1 only: Register, Log In,
             Verify Email). Once logged in they become one of the roles
             below. The label is "User", not "Visitor" or "Client".
  Admin   — system-level administrator; full platform access.
  Director — company workspace owner; manages team and subscription.
  Accountant — financial reviewer; has contract dates in the workspace.
  Employee — invoice submitter; basic workspace member.
  Personal — solo user with a personal workspace; no team features.
  System  — non-human background actor representing automated processes
             (cron jobs, async triggers). Stick figure labeled "System".
             Used where no human initiates the action. NOT a technology
             name — never label it "node-cron", "Cron Job", or "Server".

External systems (right side of boundary):
  BrevoEmailService — transactional email delivery (Sprint 1 + 4).
  GroqVisionAPI     — OCR extraction provider (Sprint 3 only).
  GroqChatAPI       — AI chat completion API (Sprint 5 only).
  Browser           — native PWA install API (Sprint 5 UC-5.11 only).

## 1.3 What must NEVER appear

- "Visitor", "WorkspaceMember", "Scheduler", "Client", "Frontend",
  "Backend", "Server", "API", "Database", "PostgreSQL", "React App"
  as actor labels anywhere. These are not roles in this system.
- "Log In", "Authenticate", or "Verify Token" use cases inside Sprint
  2–5 boundaries. Authentication is a precondition, not a use case in
  those sprints.
- "requireActiveSubscription" or "checkSubscription" as a use case
  ellipse anywhere. Subscription-gating is a precondition, described
  in UC notes, never drawn as a separate UC.
- HTTP verbs, route paths, table names, SQL, or npm package names
  (Multer, bcrypt, node-cron, Socket.io, Tesseract) in any label.
- Use cases from other sprints inside the current sprint's boundary.

## 1.4 Subscription-gated use cases (Sprint 2 onward)

The following use cases are guarded by requireActiveSubscription.
Mention it in the description but do NOT draw a separate UC for it:
  Upload Invoice, Attach Document, Update Invoice Metadata,
  Delete Draft Invoice, Delete Document, Submit for Review,
  Resubmit Invoice, Approve/Reject/Pay/Archive Invoice.

## 1.5 Drawing shared use cases with multiple actors

When Employee, Director, Accountant, and Personal ALL have identical
access to the same use case, draw four separate association lines —
one from each actor to the UC ellipse. There is no "WorkspaceMember"
parent actor in this system. When only some roles have access, draw
lines only from the qualifying roles.

# 3. SPRINT 1 USE CASE DIAGRAM
#####################################################################

Title: Sprint 1 — Authentication & Administration
Boundary label: "EASYfact — Authentication & Administration"
Sprint goal: secure multi-role authentication, workspace initialisation,
and administrator full CRUD as the technical baseline.

Actors in this diagram (left side): User, Admin, Director, Employee,
  Accountant
External (right side): BrevoEmailService

------------------------------------------------------------
3.1 Authentication Use Cases
------------------------------------------------------------

UC-1.1  Register Account
  Actors: User
  Description: unauthenticated person submits name, email, password;
  system validates uniqueness, hashes password, creates a User record,
  auto-provisions a Personal workspace + Membership, then sends a
  verification email.
  Relationships:
    UC-1.1 «include» UC-1.2  (email always sent on registration)

UC-1.2  Send Verification Email
  Actors: BrevoEmailService (external — right side)
  Description: system composes a one-time verification code email and
  dispatches it via the Brevo HTTP API.
  Included by UC-1.1. No human actor on the left for this UC.

UC-1.3  Verify Email Address
  Actors: User
  Description: user enters or clicks the verification code received
  by email; system marks the account is_verified = true.
  Note: standalone UC — happens in a separate browser session after
  registration, NOT an «extend» of UC-1.1.

UC-1.4  Log In
  Actors: User
  Description: user submits email + password; system validates
  credentials and issues a JWT access token + refresh token.

UC-1.5  Refresh Access Token
  Actors: User (client-initiated silently when access token expires)
  Description: client sends the refresh token; server validates it and
  issues a new access token. Has its own endpoint — distinct UC.

UC-1.6  Log Out
  Actors: Admin, Director, Accountant, Employee, Personal
  Description: client discards tokens; session ends.
  Note: draw five individual association lines — one from each role.

UC-1.7  Switch Active Workspace
  Actors: Director, Employee, Accountant, Personal
  Description: authenticated user selects a different workspace from
  the workspace switcher; context changes without re-login.
  Note: Admin does not switch workspaces (system-level access only).

------------------------------------------------------------
3.2 Admin — Company CRUD
------------------------------------------------------------

UC-1.8   Create Company
  Actors: Admin
  Description: Admin creates a new company record (name, industry,
  address, logo, etc.) in the platform.

UC-1.9   View Company List
  Actors: Admin
  Description: Admin retrieves the paginated list of all registered
  companies.

UC-1.10  View Company Details
  Actors: Admin
  Description: Admin views a single company's full profile.
  Relationships:
    UC-1.10 «extend» UC-1.9  [condition: Admin selects a company
                               from the list]

UC-1.11  Update Company
  Actors: Admin
  Description: Admin edits a company's name, industry, address, or logo.

UC-1.12  Delete Company
  Actors: Admin
  Description: Admin permanently removes a company and cascades
  deletion to its workspaces and memberships.

------------------------------------------------------------
3.3 Admin — User CRUD
------------------------------------------------------------

UC-1.13  Create User
  Actors: Admin
  Description: Admin creates a user account directly, bypassing
  self-registration, and assigns an initial role.

UC-1.14  View User List
  Actors: Admin
  Description: Admin retrieves all users; filterable by role and
  verification status.

UC-1.15  View User Details
  Actors: Admin
  Description: Admin views a single user's profile and all their
  workspace memberships.
  Relationships:
    UC-1.15 «extend» UC-1.14  [condition: Admin selects a user
                                from the list]

UC-1.16  Update User Role
  Actors: Admin
  Description: Admin changes a user's role assignment within a workspace.

UC-1.17  Delete User
  Actors: Admin
  Description: Admin permanently removes a user account and all
  associated memberships.

------------------------------------------------------------
3.4 Workspace Initialisation
------------------------------------------------------------

UC-1.18  Create Company Workspace
  Actors: Director
  Description: Director creates a new company workspace and links it
  to a company record; system generates a unique 6-character join code.
  Relationships:
    UC-1.18 «include» UC-1.19  (join code always generated on creation)

UC-1.19  Generate Join Code
  Actors: (system-internal — no actor association line on the left)
  Description: system generates and stores a unique alphanumeric join
  code for the new workspace. Reached only via «include» from UC-1.18.

UC-1.20  View Workspace Join Code
  Actors: Director
  Description: Director retrieves and copies the join code (from Team
  Management or Settings) to share with prospective members.

UC-1.21  Join Company Workspace
  Actors: Employee, Accountant
  Description: user submits the join code and selects a role; system
  creates a Membership record with status pending_approval; Director
  reviews it in Sprint 2 Team Management UCs.
  Note: Personal users do not join company workspaces — their Personal
  workspace is auto-created at registration (UC-1.1).

------------------------------------------------------------
Sprint 1 — Relationships summary
------------------------------------------------------------
  UC-1.1  «include» UC-1.2
  UC-1.10 «extend»  UC-1.9   [Admin selects company from list]
  UC-1.15 «extend»  UC-1.14  [Admin selects user from list]
  UC-1.18 «include» UC-1.19

No other «include»/«extend» relationships in Sprint 1.

# 4. SPRINT 2 USE CASE DIAGRAM
#####################################################################

Title: Sprint 2 — Invoice Management, Documents & Team Management
Boundary label: "EASYfact — Invoice & Document Management / Team Management"
Sprint goal: end-to-end invoice ingestion, document storage, status
visibility, and team membership management.

Actors in this diagram (left side): Employee, Director, Accountant,
  Personal
No external systems in Sprint 2.

------------------------------------------------------------
4.1 Invoice CRUD
------------------------------------------------------------

UC-2.1  Upload Invoice  (Create)
  Actors: Employee, Director, Personal
  Description: user submits invoice metadata (vendor name, amount,
  invoice date, due date, currency, category, notes) and attaches a
  PDF or image file (JPEG, PNG, TIFF, WebP, max 10 MB); system creates
  an Invoice record and stores the file.
  Subscription-gated: yes.
  Note: Accountant does NOT upload invoices (read-only for creation).
  Draw association lines from Employee, Director, Personal only.

UC-2.2  View Invoice List  (Read)
  Actors: Employee, Director, Accountant, Personal
  Description: user retrieves a paginated, filterable list of invoices.
  Filters: vendor name keyword, date range, status. Role-scoped:
  Employee and Personal see own invoices; Accountant and Director see
  all workspace invoices.
  Draw four individual association lines — one from each role.

UC-2.3  View Invoice Details  (Read)
  Actors: Employee, Director, Accountant, Personal
  Description: user views a single invoice's full record: metadata,
  current status badge, attached documents panel, status history
  stepper, and extracted-fields section.
  Relationships:
    UC-2.3 «extend» UC-2.2  [condition: user selects an invoice
                              from the list]

UC-2.4  Update Invoice Metadata  (Update)
  Actors: Employee, Director, Personal
  Description: user edits the invoice's vendor name, amount, dates,
  category, currency, or notes before submission.
  Precondition: invoice.current_status == 'draft'.
  Subscription-gated: yes.

UC-2.5  Delete Draft Invoice  (Delete)
  Actors: Employee, Director, Personal
  Description: user permanently deletes an invoice still in draft status.
  Precondition: invoice.current_status == 'draft'.
  Subscription-gated: yes.

------------------------------------------------------------
4.2 Document CRUD
------------------------------------------------------------

UC-2.6  Attach Document to Invoice  (Create)
  Actors: Employee, Director, Accountant, Personal
  Description: user uploads a supporting file (PDF, JPEG, PNG, TIFF,
  WebP, max 10 MB) and attaches it to an existing invoice.
  Subscription-gated: yes.
  Draw four individual association lines.

UC-2.7  View Invoice Documents  (Read)
  Actors: Employee, Director, Accountant, Personal
  Description: user lists all document attachments for a given invoice,
  including file type icon and upload metadata.
  Draw four individual association lines.

UC-2.8  Download Document  (Read)
  Actors: Employee, Director, Accountant, Personal
  Description: user downloads the original file for a document attachment.
  Relationships:
    UC-2.8 «extend» UC-2.7  [condition: user selects a document
                              to download]

UC-2.9  Delete Document  (Delete)
  Actors: Employee, Director, Personal
  Description: user removes a document attachment from an invoice.
  Subscription-gated: yes.
  Note: Accountant is excluded from document deletion (route guard
  permits Admin, Director, Employee, Personal only — not Accountant).
  Draw three association lines: Employee, Director, Personal.

------------------------------------------------------------
4.3 Team Management CRUD (Director only)
------------------------------------------------------------

UC-2.10  View Team Members  (Read)
  Actors: Director
  Description: Director views all workspace members grouped by role,
  with join date, contract_start, contract_end, and membership status
  (active / pending / leave_requested / renewal_requested).

UC-2.11  Accept Join Request  (Update)
  Actors: Director
  Description: Director approves a pending join request; sets
  contract_start and contract_end; membership status becomes active.
  Relationships:
    UC-2.11 «extend» UC-2.10  [condition: Director acts on a pending
                                join request in the member list]

UC-2.12  Reject Join Request  (Update)
  Actors: Director
  Description: Director denies a pending join request; no active
  membership is created.
  Relationships:
    UC-2.12 «extend» UC-2.10  [condition: Director acts on a pending
                                join request in the member list]

UC-2.13  Accept Leave Request  (Update)
  Actors: Director
  Description: Director approves a leave request; member is immediately
  removed from the workspace.
  Relationships:
    UC-2.13 «extend» UC-2.10  [condition: Director acts on a pending
                                leave request in the member list]

UC-2.14  Reject Leave Request  (Update)
  Actors: Director
  Description: Director denies a leave request; member remains active.
  Relationships:
    UC-2.14 «extend» UC-2.10  [condition: Director acts on a pending
                                leave request in the member list]

UC-2.15  Accept Renewal Request  (Update)
  Actors: Director
  Description: Director approves a contract renewal; extends the
  member's contract_end to a new date.
  Relationships:
    UC-2.15 «extend» UC-2.10  [condition: Director acts on a pending
                                renewal request in the member list]

UC-2.16  Reject Renewal Request  (Update)
  Actors: Director
  Description: Director denies a contract renewal; contract_end stays
  unchanged.
  Relationships:
    UC-2.16 «extend» UC-2.10  [condition: Director acts on a pending
                                renewal request in the member list]

UC-2.17  Remove Member  (Delete)
  Actors: Director
  Description: Director forcibly removes an active member from the
  workspace without a leave request.
  Relationships:
    UC-2.17 «extend» UC-2.10  [condition: Director removes an active
                                member from the list]

# 5. SPRINT 3 USE CASE DIAGRAM
#####################################################################

Title: Sprint 3 — OCR Processing, Invoice Workflow & Subscription
Boundary label: "EASYfact — OCR Extraction / Invoice Workflow / Subscription"
Sprint goal: OCR with confidence scoring, full invoice status-machine
workflow, and subscription plan management with automatic expiry.

Actors in this diagram (left side): Director, Accountant, Employee,
  Personal, System
External (right side): GroqVisionAPI

System = stick figure labeled "System"; represents the automated
background processes (async OCR trigger, hourly expiry cron job).

------------------------------------------------------------
5.1 OCR Processing (System-initiated after invoice upload)
------------------------------------------------------------

UC-3.1  Trigger OCR Processing  (Create — system-initiated)
  Actors: System (left side), GroqVisionAPI (external — right side)
  Description: after an invoice file is stored (Sprint 2 UC-2.1), the
  system asynchronously sends the document image to GroqVisionAPI;
  receives per-field extraction results with confidence scores.
  Relationships:
    UC-3.1 «include» UC-3.2  (extracted fields always persisted on success)
  Note: GroqVisionAPI is the documented OCR provider per the academic
  report. No human role initiates this UC.

UC-3.2  Persist Extracted Fields  (Create — system-internal)
  Actors: (system-internal — no actor association line on the left)
  Description: system saves each extracted field (invoice_number,
  invoice_date, due_date, total_amount, tax_amount, supplier_name)
  with its confidence score.
  Included by UC-3.1. Draw with no actor line.

UC-3.3  View Extracted Fields  (Read)
  Actors: Employee, Director, Accountant, Personal
  Description: user views OCR-extracted field values and per-field
  confidence scores on the invoice detail page. Low-confidence fields
  are flagged needs_review.
  Draw four individual association lines.

UC-3.4  Update Extracted Field  (Update — manual correction)
  Actors: Employee, Director, Accountant
  Description: user manually overrides an extracted field value when
  the confidence score is low or the value is incorrect.
  Subscription-gated: yes.
  Note: Personal is excluded from editing extracted fields.
  Relationships:
    UC-3.4 «extend» UC-3.3  [condition: user edits a flagged field]

UC-3.5  Sync All Fields from OCR  (Update — bulk populate)
  Actors: Employee, Director, Accountant
  Description: user clicks "Sync from OCR" to overwrite all invoice
  metadata fields with the latest OCR extraction values in one action.
  Relationships:
    UC-3.5 «extend» UC-3.3  [condition: user clicks "Sync from OCR"]

------------------------------------------------------------
5.2 Invoice Status Workflow (all are Update on invoice.current_status)
------------------------------------------------------------

UC-3.6  Submit Invoice for Review
        (Update: draft → pending_review)
  Actors: Employee, Director, Personal
  Description: user submits a draft invoice into the accountant review
  queue; status transitions to pending_review.
  Subscription-gated: yes.

UC-3.7  Resubmit Rejected Invoice
        (Update: rejected → pending_review)
  Actors: Employee, Director, Personal
  Description: user corrects a previously rejected invoice and
  resubmits it; status transitions back to pending_review.
  Subscription-gated: yes.
  Relationships:
    UC-3.7 «extend» UC-3.6  [condition: invoice.current_status
                              == 'rejected']

UC-3.8  Approve Invoice
        (Update: pending_review → approved)
  Actors: Accountant
  Description: Accountant marks a pending invoice as approved after
  validating the extracted fields and supporting documents.

UC-3.9  Reject Invoice with Notes
        (Update: pending_review → rejected)
  Actors: Accountant
  Description: Accountant rejects an invoice with mandatory rejection
  notes; status becomes 'rejected'.

UC-3.10  Mark Invoice as Paid
         (Update: approved → paid)
  Actors: Director
  Description: Director confirms payment has been made; status
  transitions to 'paid'.

UC-3.11  Mark Invoice as Archived
         (Update: paid → archived)
  Actors: Director
  Description: Director archives a paid invoice for long-term record
  keeping; status transitions to 'archived'.

UC-3.12  View Status History  (Read)
  Actors: Employee, Director, Accountant, Personal
  Description: user views the full chronological status-change log for
  an invoice: who changed it, when, to which status, and any rejection
  notes.
  Draw four individual association lines.

------------------------------------------------------------
5.3 Subscription CRUD
------------------------------------------------------------

UC-3.13  Subscribe to Plan  (Create — first subscription)
  Actors: Director
  Description: Director selects a plan (company: Starter €49, Business
  €149, Professional €349, Enterprise €999; personal: Free €0, Basic €9,
  Plus €19, Premium €39) and initiates payment; a Subscription record
  is created with status active and a current_period_end date.

UC-3.14  View Current Subscription  (Read)
  Actors: Director
  Description: Director views their active plan name, feature flags
  (has_chat, has_dm, can_create_channels), current_period_end, and
  payment status.

UC-3.15  Upgrade Subscription Plan  (Update — plan change)
  Actors: Director
  Description: Director switches to a different plan; unused days on
  the current company plan are pro-rated as credit (personal plans:
  no credit, full price charged); Subscription record updated to the
  new plan and new billing period.

UC-3.16  Renew Subscription  (Update — period extension)
  Actors: Director
  Description: Director extends an active or expired subscription for
  another billing period on the same plan.

UC-3.17  Detect Expired Subscriptions  (System — hourly cron)
  Actors: System
  Description: an automated hourly job finds all subscriptions where
  current_period_end < NOW() and status != 'expired'; marks them as
  expired; triggers a notification to the workspace Director.
  Relationships:
    UC-3.17 «include» UC-3.18  (lock always follows expiry detection)

UC-3.18  Lock Workspace for Non-Directors  (System — access block)
  Actors: System (triggers the lock)
  + Employee, Accountant (secondary associations — they experience the
    lock screen)
  Description: when a workspace subscription expires, non-director
  members are presented with the SubscriptionLock screen on their next
  request; Director sees a renewal banner. Included by UC-3.17.
  Note: draw System on the left with the «include» arrow from UC-3.17;
  also draw Employee and Accountant on the left with association lines
  to UC-3.18 to show they are the affected parties (experiencing the
  lock, not triggering it).

# 8. RENDERING RULES (apply to all 5 sprint diagrams)
#####################################################################

1.  One system boundary rectangle per sprint diagram. Title shown
    inside or above the rectangle.

2.  Human actors (User, Admin, Director, Accountant, Employee,
    Personal, System) always on the LEFT of the boundary.
    External systems (BrevoEmailService, GroqVisionAPI, GroqChatAPI,
    Browser) always on the RIGHT.

3.  No "WorkspaceMember" actor exists. When multiple roles share a
    use case, draw individual association lines from each qualifying
    role. Four lines to the same ellipse is correct and expected.

4.  "User" actor (Sprint 1 only) = unauthenticated person doing
    Register / Log In / Verify Email. In Sprint 2–5 all actors are
    already authenticated; "User" does not appear.

5.  "System" actor = automated background processes (OCR trigger,
    expiry cron). Appears only in Sprint 3 (UC-3.1, UC-3.17, UC-3.18)
    and Sprint 4 (UC-4.9). Never in Sprint 1, 2, or 5.

6.  «include» arrow direction: FROM base UC → TO included UC.
    Label: «include». The included UC executes every time.

7.  «extend» arrow direction: FROM extending UC → TO base UC.
    Label: «extend». Add the condition in square brackets on the arrow.

8.  System-internal UCs with no human initiator (UC-1.19, UC-3.2,
    UC-5.10): draw inside the boundary as normal ellipses but with NO
    actor association line. Reached only via «include» arrows.

9.  External system actors (BrevoEmailService in UC-1.2 and UC-4.10,
    GroqVisionAPI in UC-3.1): draw the association line from the
    external actor to the use case they participate in executing.

10. Never draw "Log In", "Authenticate", or any auth check as a use
    case in Sprint 2–5 diagrams. Authentication is a precondition.

11. Never draw any subscription check as a use case in any sprint.
    It is always a precondition described in the UC text only.

12. Use case label style: short, verb-first, title-case, no tech names.
    Correct: "Upload Invoice", "Mark as Paid", "Delete Document".
    Wrong: "POST /invoices", "updateStatus()", "requireSubscription".

13. Client-side-only UCs (UC-4.3–UC-4.7 export, UC-5.11 PWA):
    fully valid use cases even with no backend. Keep each export
    format separate — do NOT collapse into one "Export Report" UC.

14. UC-5.9 (Typing Indicator): draw as one UC. Do NOT split into
    "Start Typing" and "Stop Typing". No Entity involved.

15. GroqVisionAPI: Sprint 3 only (UC-3.1).
    GroqChatAPI: Sprint 5 only (UC-5.1).
    Browser: Sprint 5 only (UC-5.11).
    BrevoEmailService: Sprint 1 (UC-1.2) and Sprint 4 (UC-4.10) only.
    Never place an external system in the wrong sprint diagram.

16. Admin: Sprint 1 only. Not in Sprint 2–5 diagrams.

17. Status-workflow UCs (UC-3.6–UC-3.11): keep each as a separate UC.
    Never merge Approve and Reject — different actors, different
    preconditions, different status transitions.

18. Browser in UC-5.11 is an EXTERNAL SYSTEM, not a human actor.
    Draw it on the right side of the boundary like GroqChatAPI.
