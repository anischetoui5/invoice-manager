#####################################################################
# 1. UML STYLE
#####################################################################

## 1.1 Abstraction level chosen: layered Boundary–Control–Entity (BCE),
adapted to the project's real 3-tier code structure.

The codebase is NOT a classic ORM/repository architecture. Every module
under server/src/modules/<name>/ follows exactly this chain:

    Route (Express path) -> Controller (req/res only) -> Service
    (business logic + raw SQL via a shared pg Pool, transactions with
    BEGIN/COMMIT/ROLLBACK when multiple writes must be atomic) -> rows

There is no separate Repository/DAO/Model class anywhere in the code —
the Service function IS the data-access layer (it issues the SQL
itself). Inventing a "Repository" lifeline would therefore NOT be
faithful to the implementation. Instead, the persisted object that a
Service reads or writes is represented as an **Entity** participant
(e.g. :Invoice, :User, :Membership). The Entity lifeline does not show
SQL — it receives abstract operation calls such as "persist()",
"findById()", "updateStatus()" and is understood to represent the
row(s) in the corresponding table.

This yields a 4-role BCE-derived stereotype set:

| Stereotype   | Maps to (code)                                   | Shown as |
|--------------|---------------------------------------------------|----------|
| «boundary»   | a React page/component (client/src/app/pages/*)  | :PageName |
| «control»    | an Express controller (*.controller.js)           | :XController |
| «control»    | a Service module (*.service.js) — business logic  | :XService |
| «entity»     | the persisted domain object/table the Service acts on | :EntityName |

Both the Controller and the Service are tagged «control» (UML allows
multiple control objects collaborating). They are kept as two distinct
lifelines — never merged — because the code itself separates HTTP
concerns (status codes, req/res shape, validation of presence of
fields) from business rules (authorization logic beyond role-gating,
calculations, multi-table transactions, notification/activity side
effects). Merging them would hide real branching logic that exists in
the Service layer and would not match the implementation.

## 1.2 Participants — naming convention

- Boundary lifelines are named after the actual frontend page/component
  that fires the request, e.g. `:LoginPage`, `:UploadInvoicePage`,
  `:TeamManagementPage`. Never use "React Frontend", "Client", or "UI".
- Control (Controller) lifelines are named `:<Module>Controller`,
  matching the real file, e.g. `:AuthController`, `:InvoicesController`,
  `:OcrController`. Never use "Express API", "Router", or "REST API".
- Control (Service) lifelines are named `:<Module>Service`, e.g.
  `:AuthService`, `:InvoicesService`, `:OcrService`,
  `:InvitationsService`, `:SubscriptionService`, `:NotificationsService`,
  `:ActivityService`, `:EmailService`.
- Entity lifelines are named after the singular domain concept, matching
  the class-diagram class names, e.g. `:User`, `:Workspace`, `:Company`,
  `:Membership`, `:Invoice`, `:Document`, `:ExtractedField`,
  `:StatusHistory`, `:Invitation`, `:Subscription`, `:Notification`,
  `:ActivityLog`. Never use "PostgreSQL", "Database", "Pool", or a table
  name in snake_case.
- The non-human background actor is `:Scheduler` (the node-cron job
  runner), never "node-cron" or "Cron Job" as a class name — keep
  "Scheduler" as the UML-facing label; node-cron is an implementation
  detail that may appear only in a side note, not as the lifeline name.
- External systems sit outside the boundary and are named for what they
  do, not their brand mechanics: `:BrevoEmailService` (email delivery)
  and, per the academic report's documented architecture for Sprint 3,
  `:GroqVisionAPI` (OCR extraction). These are the only two external
  participants permitted in Sprints 1–3.

## 1.3 What must NEVER appear

- No SQL keywords (SELECT, INSERT, UPDATE, DELETE, JOIN, WHERE, BEGIN,
  COMMIT) in any message label, anywhere.
- No technology/framework names as lifelines: "React Frontend",
  "Express API", "PostgreSQL", "Axios", "Multer", "node-cron",
  "pg Pool", "Tesseract", "bcrypt".
- No raw HTTP route strings as message labels (no "POST /api/auth/login").
  Use the equivalent operation call instead: `login(email, password)`.
- No middleware shown as its own lifeline in ordinary CRUD diagrams
  (see §1.4). Middleware logic is folded into a guard condition.
- No file-system or disk-path detail (e.g. "uploads/", "fs.writeFile").
  Use "store document" as an abstract Entity-creation step.

## 1.4 Middleware policy

The code has 4 middleware functions: `authenticate` (JWT check),
`authorizeInWorkspace(...)` (role gate), `authorizeAdmin`, and
`requireActiveSubscription`. None of these has meaningful internal
collaboration worth a dedicated lifeline in every diagram — showing 4
extra lifelines on 30+ diagrams, all doing the same JWT/role check,
would clutter the diagrams without adding insight, and would violate
the "keep it useful for an academic reader" goal.

Rule applied throughout this document:
- In the **Global Sequence Diagram** and in **User Login** /
  **User Registration**, show authentication explicitly as a self-call
  on the Controller: `validateSession()` / JWT issuance, because
  authentication IS the subject of those diagrams.
- In every other diagram, represent the middleware chain as a single
  **opt/alt guard fragment** at the very top of the diagram, e.g.:
    `opt [session valid AND role authorized]`
  with a one-line note: "JWT + role check performed by AuthGuard
  (authenticate + authorizeInWorkspace); diagram continues only on
  success." Do not draw a separate `:AuthGuard` lifeline unless a
  diagram's Notes section explicitly says to.
- `requireActiveSubscription` is shown the same way, folded into the
  same guard fragment, only for the diagrams explicitly marked
  "subscription-gated" below (Upload Invoice, Update Invoice Status,
  Update Invoice, Delete Invoice, OCR field edits).

## 1.5 HTTP requests and status codes

- Do not write HTTP verbs/paths as messages. Translate each REST call
  into the equivalent UML operation call on the Controller, named after
  the actual controller export, e.g. `createInvoice(payload)`,
  `updateInvoiceStatus(id, status, comment)`.
- Return messages MAY carry a short semantic outcome label including the
  status code where it clarifies the result of an alt branch, e.g.
  `return 201: invoice created` / `return 403: role not authorized`.
  This is acceptable because it communicates outcome, not protocol — but
  keep it on the **return arrow only**, never on the request arrow.

## 1.6 Verdict: BCE (adapted) is the right fit

Plain BCE (only 3 stereotypes) under-represents this codebase because
Controller and Service are genuinely separate files with separate
responsibilities. A strict 3-tier "Presentation/Business/Data" model
under-represents it because there is no Data layer distinct from the
Service. The 4-role model in §1.1 (boundary / control-controller /
control-service / entity) is the smallest model that stays faithful to
every module inspected (auth, users, company, workspace, invoices,
documents, ocr, invitations, subscription, notifications, activity) and
is what the rest of this document uses uniformly.

# 2. GLOBAL SEQUENCE DIAGRAM
#####################################################################

Title: EASYfact — Global Interaction Overview (Authenticated Request Lifecycle)

Objective: show, once, the canonical path every authenticated feature
request takes through the system's layers, including the
authentication/authorization guard, so that every later diagram can
omit re-explaining it. This is the only diagram in which the guard
middleware is drawn as its own lifeline.

Participants (left to right):
1. `User` (actor — generic; stands for any of Admin/Director/Accountant/
   Employee/Personal, since the path is identical for all of them)
2. `:AppPage` «boundary» — generic stand-in for "whichever page the user
   is on" (Login/Dashboard/Invoices/etc.) — used only in this overview
   diagram; every later diagram replaces it with the specific page.
3. `:FeatureController` «control» — generic stand-in for the specific
   controller of the feature being invoked.
4. `:AuthGuard` «control» — composite representation of
   `authenticate` + `authorizeInWorkspace` (+ `requireActiveSubscription`
   when applicable).
5. `:FeatureService` «control» — generic stand-in for the specific
   service handling the business logic.
6. `:Entity` «entity» — generic stand-in for whichever domain object is
   read/written.

Responsibilities:
- `AppPage`: collects user input, fires the operation call, renders the
  result or the error toast.
- `FeatureController`: parses/validates the presence of required
  request fields, delegates to `AuthGuard`, then to `FeatureService`,
  and maps the outcome to a return value/status.
- `AuthGuard`: verifies the JWT signature/expiry; resolves the caller's
  role inside the target workspace; rejects if no membership or role
  not permitted; (optionally) rejects if the workspace's subscription
  is expired.
- `FeatureService`: executes the business rule(s), opens a transaction
  if more than one Entity must change atomically, triggers any side
  effects (Notification, ActivityLog).
- `Entity`: represents the persisted row(s); responds to find/persist/
  update calls with the resulting state.

Main flow:
1. User -> AppPage : performs an action (click/submit)
2. AppPage -> FeatureController : invoke(operation, payload)
3. FeatureController -> AuthGuard : authorize(token, workspaceId, requiredRoles)
4. AuthGuard --> FeatureController : return ok(role) | reject(reason)
5. [alt: ok] FeatureController -> FeatureService : execute(payload, role)
6. FeatureService -> Entity : find/persist/update(...)
7. Entity --> FeatureService : return entityState
8. FeatureService --> FeatureController : return result
9. FeatureController --> AppPage : return 200/201 result
10. AppPage -> User : render success state

Alternative flows:
- alt [AuthGuard rejects — no/invalid token]: AuthGuard --> FeatureController:
  return reject("401 Unauthorized") ; FeatureController --> AppPage:
  return 401 ; AppPage -> User : redirect to Login
- alt [AuthGuard rejects — role not permitted]: return reject("403
  Forbidden") ; AppPage -> User : show "Access denied"
- alt [AuthGuard rejects — subscription expired, when applicable]:
  return reject("403 SUBSCRIPTION_EXPIRED") ; AppPage -> User : show
  SubscriptionLock screen
- opt [FeatureService raises a business error, e.g. invalid state
  transition]: FeatureService --> FeatureController : return error ;
  FeatureController --> AppPage : return 400/409 ; AppPage -> User :
  show error toast

Notes for the generating AI:
- This diagram is intentionally generic. Every concrete diagram in
  Sections 3–5 below reuses this skeleton but renames AppPage,
  FeatureController, FeatureService and Entity to the concrete classes,
  and — per §1.4 — collapses AuthGuard into a single `opt` guard
  fragment instead of a full lifeline.
- Do not add Repository/DAO lifelines anywhere.
- Keep arrowheads as plain synchronous calls (filled arrowhead, solid
  line) for all controller/service/entity messages; dashed arrows for
  all `return` messages.

3.1 Diagram: User Registration
------------------------------------------------------------
Objective: register a new account, auto-provision a personal workspace,
and send a verification code.
Actor: User (unauthenticated)
Participants: :RegisterPage, :AuthController, :AuthService, :User
(entity), :Workspace (entity), :Membership (entity), :EmailService,
:BrevoEmailService (external)

Ordered messages:
1. User -> RegisterPage : submit(name, email, password)
2. RegisterPage -> AuthController : register({name, email, password})
3. AuthController -> AuthService : register(payload)
4. AuthService -> User : findByEmail(email)
5. User --> AuthService : return null (not found)
6. AuthService -> AuthService : hashPassword(password)  [self-call]
7. AuthService -> User : persist(name, email, passwordHash)
8. User --> AuthService : return newUser
9. AuthService -> Workspace : createPersonal(owner=newUser)
10. Workspace --> AuthService : return personalWorkspace
11. AuthService -> Membership : create(newUser, personalWorkspace, role=Personal)
12. Membership --> AuthService : return membership
13. AuthService -> AuthService : generateVerificationCode()  [self-call]
14. AuthService -> EmailService : sendVerificationCode(email, code)
15. EmailService -> BrevoEmailService : deliver(email, code)
16. BrevoEmailService --> EmailService : return ack
17. EmailService --> AuthService : return (fire-and-forget; failure is
    swallowed, code is logged server-side as a fallback — see Notes)
18. AuthService --> AuthController : return {userId, email}
19. AuthController --> RegisterPage : return 201 "verify your email"
20. RegisterPage -> User : navigate to Verify-Email screen

Alternative flows:
- alt [email already registered]: User --> AuthService returns existing
  row at step 5 ; AuthService --> AuthController : return error("email in
  use") ; AuthController --> RegisterPage : return 409 Conflict
- opt [Brevo delivery fails/unreachable]: EmailService --> AuthService :
  return failure (caught, not thrown) ; flow continues normally to step
  18 — the verification code is still valid and is also written to the
  server console as a fallback channel.

Notes: this is the only diagram besides Login where `AuthGuard` is
irrelevant (the user has no token yet) — do not draw it at all.

------------------------------------------------------------
3.2 Diagram: User Login and JWT Issuance
------------------------------------------------------------
Objective: authenticate credentials and issue a signed JWT.
Actor: User (unauthenticated)
Participants: :LoginPage, :AuthController, :AuthService, :User (entity)

Ordered messages:
1. User -> LoginPage : submit(email, password)
2. LoginPage -> AuthController : login({email, password})
3. AuthController -> AuthService : login(payload)
4. AuthService -> User : findByEmail(email)
5. User --> AuthService : return userRecord
6. AuthService -> AuthService : checkPassword(password, userRecord.passwordHash) [self-call]
7. AuthService -> AuthService : signJwt(userId, email, workspaceId) [self-call]
8. AuthService --> AuthController : return {token, user}
9. AuthController --> LoginPage : return 200 {token, user}
10. LoginPage -> User : store token, navigate to Dashboard

Alternative flows:
- alt [no user found OR password mismatch]: AuthService --> AuthController :
  return error("invalid credentials") ; AuthController --> LoginPage :
  return 401 Unauthorized
- alt [user.isVerified == false]: AuthService -> AuthService :
  generateVerificationCode() ; AuthService -> EmailService :
  sendVerificationCode(email, code) [fire-and-forget] ;
  AuthService --> AuthController : return error("EMAIL_NOT_VERIFIED") ;
  AuthController --> LoginPage : return 403 {code:"EMAIL_NOT_VERIFIED"} ;
  LoginPage -> User : prompt to verify, code already (re)sent

Notes: do not draw AuthGuard — there is no prior session to guard; this
diagram IS the act of creating one.

4.1 Diagram: Upload Invoice (Create)
------------------------------------------------------------
Objective: create a draft invoice, attach the uploaded file as a
Document, and asynchronously trigger OCR after responding to the client.
Actor: Employee, Director, or Personal
Participants: :UploadInvoicePage, :InvoicesController, :InvoicesService,
:Invoice (entity), :StatusHistory (entity), :ActivityService,
:NotificationsService, :DocumentsController, :DocumentsService,
:Document (entity), :OcrService

Ordered messages (two sequential HTTP calls in the real flow: first
create the invoice record, then attach the file — represent both in one
diagram since they are one user-perceived action):
1. User -> UploadInvoicePage : submit(invoiceMetadata, file)
2. UploadInvoicePage -> InvoicesController : createInvoice(metadata)
3. InvoicesController -> InvoicesService : createInvoice(metadata)
4. InvoicesService -> Subscription : checkInvoiceQuota(workspaceId)  [entity from Subscription domain]
5. Subscription --> InvoicesService : return {withinQuota: true}
6. InvoicesService -> Invoice : persist(metadata, status='draft')
7. Invoice --> InvoicesService : return invoice
8. InvoicesService -> StatusHistory : record(invoice, status='draft')

5.1 Diagram: OCR Processing and Confidence Scoring
------------------------------------------------------------
Objective: extract structured invoice fields from the uploaded document
with per-field confidence scoring, fully decoupled from the upload HTTP
response.
Actor: System (background trigger, no human actor) + GroqVisionAPI
(external, per the documented Sprint-3 architecture)
Participants: :DocumentsController (trigger), :OcrService, :Invoice
(entity), :GroqVisionAPI (external), :ExtractedField (entity),
:NotificationsService, :Notification (entity)

1. DocumentsController -> OcrService : processInvoice(invoiceId, documentPath) [fired
   after the 201 Created response was already sent — see §4.1 step 21]
2. OcrService -> Invoice : updateOcrStatus(invoiceId, 'processing')
3. OcrService -> OcrService : encodeDocumentAsBase64(documentPath) [self-call]
4. OcrService -> GroqVisionAPI : extract(base64Document, extractionPrompt)
5. GroqVisionAPI --> OcrService : return {sixFieldsJson, confidenceDescriptors}
6. OcrService -> OcrService : mapDescriptorToScore(confidenceDescriptors) [self-call]
7. OcrService -> OcrService : validateAmountAndDateFormats(sixFieldsJson) [self-call]
8. loop [for each of the six fields]
   8a. OcrService -> ExtractedField : upsert(invoiceId, fieldName, value,
       confidence, needsReview = confidence < threshold)
9. OcrService -> Invoice : updateOcrStatus(invoiceId, 'completed', avgConfidence)
10. OcrService -> NotificationsService : notify(uploader, "OCR complete")
11. NotificationsService -> Notification : persist(...)

Alternative flows:
- alt [GroqVisionAPI errors or times out]: GroqVisionAPI --> OcrService :
  return error/timeout ; OcrService -> Invoice : updateOcrStatus(invoiceId,
  'failed') ; OcrService -> NotificationsService : notify(uploader,
  "OCR failed")

Notes: no human actor and no AuthGuard fragment — this runs as an
internal background continuation, not an HTTP request.

7.4 Diagram: Send Message via WebSocket
------------------------------------------------------------
Objective: deliver a chat message to all conversation members in real
time over a persistent socket connection (not a REST call).
Actor: Director, Accountant, Employee (workspace member, plan-gated)
Participants: :ChatPage, :ChatSocketGateway «control» (the io.use() /
event-handler layer — equivalent role to a Controller, but for sockets
rather than HTTP), :ChatService, :ChatMember (entity), :ChatMessage (entity)

Connection setup (once per session, shown as a precondition note, not
repeated per message):
- ChatPage -> ChatSocketGateway : connect(jwtToken) ; ChatSocketGateway
  validates JWT at handshake (`io.use()`); rejects if invalid.
- ChatPage -> ChatSocketGateway : emit join:workspace(workspaceId)
- ChatPage -> ChatSocketGateway : emit join:conversation(conversationId)

Ordered messages:
1. User -> ChatPage : type message, press send
2. ChatPage -> ChatSocketGateway : emit message:send({conversationId, content})
3. ChatSocketGateway -> ChatService : createMessage(conversationId, senderId, content)
4. ChatService -> ChatMember : isMember(conversationId, senderId)
5. ChatMember --> ChatService : return true
6. ChatService -> ChatMessage : persist(conversationId, senderId, content)
7. ChatMessage --> ChatService : return savedMessage
8. ChatService --> ChatSocketGateway : return savedMessage
9. ChatSocketGateway -> ChatPage : broadcast message:new(savedMessage) [to all
   sockets in conv:{conversationId} room, including the sender]
10. ChatPage -> User : render new message with date separator/timestamp

Alternative flows: alt [sender not a member of the conversation] ->
ChatService --> ChatSocketGateway : throw error ; ChatSocketGateway ->
ChatPage : emit message:error({error}) ; ChatPage -> User : show inline
error, message not sent.
Notes: this is the one diagram in this document where the Controller
role is played by a socket event-handler module rather than an Express
route controller — name it `:ChatSocketGateway` consistently, never
"Socket.io" or "WebSocket Server". No HTTP status codes apply here; use
the literal event names from the code (`message:send`, `message:new`,
`message:error`) as message labels, since these ARE the documented
"operations" of this interface, not protocol noise.
Guard fragment: connection-level JWT check only (shown above); no
additional per-message AuthGuard opt fragment — the membership check at
step 4 IS the per-message authorization.

#####################################################################
# 8. DIAGRAM GENERATION RULES (apply to every diagram above and to any
#    further diagrams generated from this document)
#####################################################################

1. Never use "React Frontend", "Client", "UI", or "Browser" as a
   lifeline. Use the specific page/component name (`:LoginPage`,
   `:UploadInvoicePage`, etc.).
2. Never use "Express API", "Server", "Backend", or "REST API" as a
   lifeline. Use the specific `:XController` / `:XService` names given
   above.
3. Never use "PostgreSQL", "Database", "DB", or "Pool" as a lifeline.
   Use the specific Entity name (`:Invoice`, `:User`, `:Membership`,
   etc.) representing the persisted domain object.
4. Never write SQL (no SELECT/INSERT/UPDATE/DELETE/JOIN/WHERE/BEGIN/
   COMMIT) anywhere on a message label. Translate every database
   interaction into a verb-based operation: find/persist/update/delete/
   upsert/record, exactly as used in the diagrams above.
5. Never show Multer, bcrypt, jsonwebtoken, node-cron, Tesseract, sharp,
   pdf-poppler, axios, or any other npm package name as a lifeline or
   message label. These are implementation details folded into the
   Service's self-calls or into the `:GroqVisionAPI`/`:BrevoEmailService`
   external participants where the report's documented architecture
   requires an explicit external system.
6. Never draw the 4 Express middleware functions
   (authenticate/authorizeInWorkspace/authorizeAdmin/
   requireActiveSubscription) as individual lifelines in feature-level
   diagrams. Represent them as a single `opt`/`alt` guard fragment per
   §1.4, except in §2 (Global) and §3.1–3.2 (Registration/Login) where
   authentication is the explicit subject.
7. Always keep Controller and Service as two separate lifelines — never
   merge them, and never skip the Controller and call the Service
   directly from the Boundary.
8. Always route every database-touching action through the Entity
   lifeline that owns that table — never let a Service "directly"
   produce data without an Entity message, even for one-line lookups.
9. Use solid lines with filled arrowheads for synchronous calls; dashed
   lines with open arrowheads for return messages. Every call that has
   a meaningful result must have an explicit return message — do not
   leave calls "hanging" without a return arrow.
10. Use `alt` for mutually exclusive branches (success vs. specific
    error conditions), `opt` for an optional step that may or may not
    execute (e.g. a guard, a threshold-based notification), and `loop`
    for iteration over a collection (e.g. "for each reviewer", "for
    each of the six OCR fields"). Do not use `alt` where `opt` is
    correct (a single optional branch with no real "else" is `opt`,
    not `alt` with an empty else).
11. Return messages must be meaningful, not generic "OK": state what
    was returned (`return invoice`, `return 403 Forbidden`,
    `return {subscription, amountCharged}`), matching the messages
    given in each diagram above.
12. Keep every lifeline name byte-for-byte consistent with the class
    diagram already produced for this project (see the class names in
    Domain: Authentication & Access, Workspace & Team Management,
    Subscription Management, Invoice Management, Chat & Communication).
    Do not introduce a new spelling or synonym for an existing class
    (e.g. always `:Membership`, never `:TeamMember` or `:Assignment`).
13. Keep the BCE-derived 4-stereotype model (boundary / control-
    controller / control-service / entity) identical across literally
    every diagram in this document — do not switch to a 3-tier or a
    pure BCE model partway through; consistency across all ~64 diagrams
    (Global + 12 Sprint 1 + 10 Sprint 2 + 15 Sprint 3 + 15 Sprint 4 +
    11 Sprint 5) is required for academic coherence.
14. Where a diagram above explicitly states "Notes:", treat that note as
    a binding instruction for how to draw that specific diagram (e.g.
    the double ActivityLog write in §4.9, the "extends" relationship in
    §5.6, the lack of any Boundary/AuthGuard in cron-triggered diagrams
    §5.1 and §5.15).
15. Per the explicit decision for this project: the Sprint 3 OCR diagram
    (§5.1) shows `:GroqVisionAPI` as the external OCR provider, matching
    the academic report's documented architecture, even though the
    current codebase implements OCR locally with Tesseract.js. Do not
    silently revert to a Tesseract-based diagram. The same treatment
    applies to `:GroqChatAPI` in §7.1 (AI Guidance) — the model name
    used in the report differs from the model name in the code, but
    since no diagram message ever names a specific model version, this
    causes no actual inconsistency; do not introduce one by adding a
    model name to any message label.
16. Report-export diagrams (§6.3 PDF, §6.4 Excel, §6.5 CSV/DOCX/ZIP) and
    the PWA-install diagram (§7.11) are the only diagrams in this
    document that are intentionally Boundary-only, with zero Controller/
    Service/Entity lifelines and zero AuthGuard fragment. This is
    faithful to the implementation (export is pure client-side
    transformation of already-fetched data; PWA install is a native
    browser capability) — do not "fill in" a Controller/Service chain
    for these just to match the shape of other diagrams.
17. `:Browser` (§7.11 only) and the three external systems (`:GroqVisionAPI`
    §5.1, `:GroqChatAPI` §7.1, `:BrevoEmailService` throughout) are the
    ONLY platform/technology-named participants permitted anywhere in
    this document. No other diagram may introduce a new platform-named
    lifeline under any circumstance.
18. The real-time chat diagrams (§7.4 Send Message, §7.8 Delete Message,
    §7.9 Typing Indicator) use `:ChatSocketGateway` as the «control»
    participant in place of a Controller, because the code's real
    entry point for these three flows is a Socket.io event handler
    (`chat.socket.js`), not an Express route. Keep this substitution
    confined to exactly these three diagrams — every other chat
    diagram (§7.2, §7.3, §7.5, §7.6, §7.7, §7.10) is a normal REST call
    and must use `:ChatController` as usual, matching the real
    `chat.controller.js` / `chat.routes.js` split in the code.
19. Self-referential plan-gating checks (`getPlanFeatures`, seen in
    §7.2, §7.3, §7.6, §7.10) must be drawn as self-calls on
    `:ChatController`, never as a separate `:AuthGuard` or
    `:SubscriptionMiddleware` lifeline — this differs from the generic
    §1.4 middleware policy because, unlike `authenticate`/
    `authorizeInWorkspace`/`requireActiveSubscription` (which ARE
    separate Express middleware files), `getPlanFeatures` is genuinely
    defined inside the controller file itself in the real code.
20. Two Service-layer functions in §6.13 ("Update Company Information",
    Director self-service from Settings) and §3.9/§3.10 ("Update/Delete
    Company", Admin CRUD) call different Service functions
    (`CompanyService.updateCompany` vs. `CompanyService.adminUpdateCompany`/
    `adminCreateCompany`/`deleteCompany`) even though both operate on the
    same `:Company` entity. Keep them as two distinct diagrams with two
    distinct Service method names — never collapse them into one
    diagram, since they have different actors, different authorization
    rules, and are reached from different pages in the real application.
